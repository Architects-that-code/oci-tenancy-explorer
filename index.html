About The OCI Tenancy Explorer

Tenancy-Wide Visibility And Operational Signals

01 What This App Does

OCI Tenancy Explorer gives administrators and customers a clear view of compute instances across the tenancy, including maintenance reboot dates, reboot-related evidence, and recent status changes.

  • • Fleet-wide view: The collector discovers OCI compute instances across all subscribed regions marked as ready in the tenancy.
  • • Reboot visibility: The dashboard shows whether an instance is scheduled, completed, or not scheduled for a reboot-related maintenance event.
  • • Change tracking: The app compares the latest snapshot with the previous one so operators can quickly spot transitions.
  • • Safe browser model: OCI credentials stay on the host running the collector and are never exposed to the browser. The page keeps fleet data in memory for the current tab session instead of storing it in persistent browser storage.

02 How It Is Installed

A customer can install this application with a very small footprint. The front end is a single HTML file, and the data collector is a Python script using the OCI SDK.

  1. Install Python 3 and the OCI Python SDK.
  2. Create OCI authentication in ~/.oci/config or use instance principals on OCI compute.
  3. Run portal_server.py on the same host that will build and serve fleet_data.json.
  4. Use the portal refresh flow or run build_fleet_data.py on that same host to generate fleet_data.json.
  5. Open the dashboard locally and refresh on demand from the UI.

OCI Authentication Setup

Option 1: Local ~/.oci/config

  • Create ~/.oci/config with user, tenancy, fingerprint, region, and key_file.
  • Place the private API key at the path referenced by key_file, for example /Users/user_name/.oci/oci_api_key.pem.
  • Upload the matching public key to the OCI user in the OCI Console under API Keys.

03 Live OCI Refresh Flow

The dashboard supports a controlled one-click refresh model that is safe for customer environments and easy for operators to understand.

  • • Refresh button: The user clicks Refresh Data inside the portal.
  • • Local endpoint: portal_server.py starts the collector in the background and streams live log output to the modal.
  • • JSON rebuild: The collector rewrites fleet_data.json with the newest tenancy snapshot.
  • • Browser sync: After fleet_data.json is updated on the host, the SPA reloads that refreshed snapshot into memory and updates the grid so added, removed, or changed rows are reflected on the next load.

04 Status Meanings

The app compares the latest OCI snapshot with the previous one to show the current reboot state in a way that is readable for both engineers and business users.

SCHED

A maintenance reboot date is currently present in OCI. The dashboard marks the instance as Scheduled.

COMP

The previous snapshot showed a maintenance reboot date, but the current snapshot no longer does. The dashboard marks the instance as Completed.

STBY

No maintenance reboot signal is currently reported by OCI for the instance. The dashboard marks it as Not Scheduled.

05 Change Signals

The dashboard highlights what changed since the previous collector run so a reboot-related transition is easy to see immediately.

  • • Signal column: Shows Stable or a transition such as Not Scheduled → Scheduled.
  • • Changed stat: Counts how many rows changed in the current filtered view.
  • • Customer card highlight: Fleet cards glow blue when any instance in that customer changed.

06 OCI Data Sources

The maintenance and reboot fields shown in the grid are based on direct OCI API data and snapshot comparison logic.

Identity Service

Used to list accessible compartments and subscribed regions. This is how the collector knows which compartments and regions to scan.

Compute Service

The collector calls OCI Compute to list instances and active instance maintenance events per compartment and region. When an InstanceMaintenanceEvent exists, it becomes the primary source for Maintenance_UTC, Maintenance_Action, and Maintenance_Event_Status, with instance metadata kept as a fallback.

OS Management Hub

When available, the collector reads managed instance boot data to populate Last_Reboot_UTC and set Reboot_Evidence to show that the reboot timestamp came from OS Management Hub.

Availability Configuration

The collector also reads each instance availability configuration so the grid can show Live_Migration_Preference. That helps distinguish reboot-required maintenance from maintenance that OCI may handle with live migration or no customer action.

How Maintenance Dates Are Determined

Step 1

The collector lists compute instances and then checks OCI InstanceMaintenanceEvent records for those same compartments and regions.

Step 2

If an active maintenance event exists, the collector uses the event timing and action as the primary maintenance source. If no event is present, it falls back to the instance field time_maintenance_reboot_due.

Step 3

If the maintenance source, action, or schedule changes between refreshes, the dashboard raises a change signal. If a reboot schedule disappears later, the app marks the instance as Completed.

Timezones And Evidence

UTC (Universal)

Primary timestamp reference returned by OCI and used for change tracking.

IST (India)

Automated offset: +5:30.

EDT (Eastern)

Automated offset: -4:00.

Reboot Evidence

Shows whether the displayed reboot context came from OCI maintenance data, OS Management Hub boot data, or prior snapshot comparison logic.

Local OCI Config Setup

Use ~/.oci/config With An API Key

Use this path when you are running the portal from a laptop, admin workstation, or another local host.

1

Create The OCI Folder

Run mkdir -p ~/.oci and chmod 700 ~/.oci on the same host that will run portal_server.py.

2

Generate Or Place The API Key

Generate an OCI signing key pair locally, or place the approved private key file on disk, for example ~/.oci/oci_api_key.pem.

3

Upload The Public Key

Upload the matching public key in the OCI Console under your user profile and copy the fingerprint OCI returns.

4

Create ~/.oci/config

Populate user, tenancy, fingerprint, region, and key_file, then run chmod 600 ~/.oci/config.

Instance Principal Setup

Run On OCI Compute Without A Local API Key

Use this path when the portal and collector run on an OCI compute instance and you want the host itself to authenticate to OCI.

1

Run On OCI Compute

Deploy the portal on an OCI compute instance. This same host should also run build_fleet_data.py and serve fleet_data.json.

2

Create A Dynamic Group

Place the compute instance in a dynamic group so OCI can recognize it as an authorized principal.

3

Grant Read Policies

Create IAM policies that let the dynamic group inspect compartments and read the OCI services this app queries.

4

Start With Instance Principal Auth

Launch the app using python3 portal_server.py --auth instance_principal. The refresh modal will then run the collector with the same auth method.
